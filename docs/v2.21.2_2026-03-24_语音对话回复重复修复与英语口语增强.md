## 任务目标

修复语音持续对话模式下 AI 回复内容重复的严重 Bug，同时优化英语口语对话工具和 TTS 文本预处理。

## 问题诊断与分析

### 问题 1：语音模式 AI 回复重复

**现象**：持续对话模式下，AI 几乎每条回复都出现两遍，例如"草莓要的...🍓🌿草莓要的...🍓🌿"。第一条消息出现两种风格拼接（长正式回复 + 短口语回复）。

**原因**：`_on_auto_send_timeout()` 在语音模式下同时发射了两个信号：
- `speech_recognized.emit(text, True)` → `_on_speech_recognized` → `message_sent.emit()` → `Agent.chat()` (标准路径)
- `speech_recognized_with_prompt.emit(text, True)` → `_on_speech_recognized_with_prompt` → `voice_message_sent.emit()` → `Agent.chat_voice()` (CFTA快速路径)

两个路径并行运行，各自生成一份回复拼接在一起显示。

### 问题 2：英语对话输出混杂中文

**现象**：英语口语练习中 AI 回复包含中文翻译和解释，影响沉浸式练习体验。

**原因**：系统 prompt 未严格限制纯英语输出，且无后处理过滤机制。

### 问题 3：英语对话 Schema 错误

**现象**：调用 `english_conversation_start_conversation` 时 Deepseek API 返回 `Invalid schema: true is not of type "array"`。

**原因**：`get_actions()` 在参数字典中写了 `"required": True/False`，但 `BaseTool.get_schema()` 要求通过 `required_params` 列表声明必填参数。

### 问题 4：TTS 朗读标点符号

**现象**：TTS 播放时朗读出标点符号和 Emoji 符号。

**原因**：文本预处理未移除标点符号，仅移除了部分 Emoji。

## 解决方案与实施

### 1. 修复语音模式回复重复（关键修复）

**文件**: `src/ui/main_window.py`

在 `_on_speech_recognized()` 中添加语音模式分支：语音模式下只负责 UI 显示（添加用户消息气泡），不再发射 `message_sent` 信号。AI 处理完全由 `voice_message_sent` 信号（CFTA 路径）接管。

```python
def _on_speech_recognized(self, text, is_voice_mode=False):
    self._chat_widget.add_user_message(text)
    self._input_edit.clear()
    
    # 【关键修复】语音模式下只负责 UI 显示
    if is_voice_mode:
        self._set_thinking_state(True)
        return
    
    # 非语音模式：正常发出信号
    ...
```

## 技术实现细节

### 信号链修复前后对比

```
修复前（两条路径并行）:
speech_recognized → message_sent → Agent.chat()         [标准路径]
speech_recognized_with_prompt → voice_message_sent → chat_voice() [CFTA路径]
→ 两份回复拼接

修复后（单一路径）:
speech_recognized → UI显示（不触发AI处理）
speech_recognized_with_prompt → voice_message_sent → chat_voice() [仅CFTA路径]
→ 单份回复
```

### 流式 TTS 降级机制

```
流式播放中:
  speak("新文本")
    → is_streaming_active == True
    → 自动降级为 enqueue("新文本")
    → 追加到播放队列（不打断当前播放）

非流式模式:
  speak("新文本")
    → is_streaming_active == False
    → 清空队列 + 停止当前 + 立即播放
```

## 经验教训

### 1. 信号链必须是单一路径

语音模式下同一条消息不应触发多个处理路径。CFTA 引入 `voice_message_sent` 信号后，必须确保 `_on_speech_recognized` 不再发射 `message_sent`，否则会导致双重处理。

**教训**: 新增信号路径时必须同步屏蔽旧路径，避免并行触发。

### 2. LLM 纯语言约束需要双重保障

仅靠系统 prompt 约束 LLM 输出语言不可靠，需要配合后处理过滤（如 `_filter_chinese_content()`）作为兜底。

**教训**: 对 LLM 输出格式的硬性要求应采用 prompt + 后处理双重机制。
