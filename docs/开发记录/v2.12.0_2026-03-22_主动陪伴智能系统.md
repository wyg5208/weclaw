## 任务目标

实现 WeClaw 的核心差异化功能——**主动陪伴智能系统 (Life Companion Intelligence)**，使 AI 从被动工具执行者进化为主动关怀用户的生活伙伴。

### 核心目标
1. **主动关怀**：AI 能够根据用户档案和行为分析，主动发起关怀交互
2. **用户档案管理**：建立完整的用户画像系统，包括家庭成员、社交关系
3. **情绪感知**：通过关键词匹配检测用户情绪状态
4. **防骚扰机制**：精细化的冷却管理，确保关怀不会变成打扰
5. **渐进式建档**：分阶段自然收集用户信息

#### 2. 关怀主题注册表 (`src/core/companion_topics.py`) — 新建，438行

关怀系统的知识库，定义了所有关怀主题和行为规则。

**核心数据结构**：
```python
@dataclass
class CareTopic:
    """关怀主题定义"""
    topic_id: str
    name: str
    description: str
    trigger_conditions: List[str]
    message_templates: List[str]
    cooldown_hours: int
    priority: int

@dataclass
class OnboardingStep:
    """渐进建档步骤"""
    step_id: int
    topic: str
    prompt: str
    required_fields: List[str]

@dataclass
class InferenceRule:
    """行为推断规则"""
    rule_id: str
    trigger_pattern: str
    inferred_attribute: str
    confidence: float
```

**预定义内容**：
- **15个关怀主题**：CareTopicRegistry 管理的预定义主题集合
- **5步渐进建档序列** (ONBOARDING_SEQUENCE)：分阶段自然收集用户信息
- **5条行为推断规则** (INFERENCE_RULES)：从用户行为中自动推断用户属性
- **情绪关键词表** (MOOD_KEYWORDS)：用于情绪检测的关键词词典
- **用户主动触发关键词** (USER_CARE_REQUEST_KEYWORDS)：用户主动请求关怀的关键词

#### 3. CompanionEngine 陪伴引擎 (`src/core/companion_engine.py`) — 新建，1408行

整个系统的核心调度器，协调所有子模块完成关怀流程。

**子模块组成**：

| 子模块 | 职责 | 关键特性 |
|--------|------|---------|
| `CooldownManager` | 防骚扰冷却管理 | 每日预算5次、连续限制2次、拒绝惩罚、asyncio.Lock 交互锁、动态预算调整 |
| `MoodDetector` | 情绪检测 | 关键词匹配，支持 positive/negative/neutral/stressed/tired 五种状态 |
| `OpportunityDetector` | 时机检测 | 监听 EventBus 事件，检测适合发起关怀的时机 |
| `InteractionOrchestrator` | 交互编排器 | 冷却检查→用户状态→消息构建→事件发布→结果记录 |
| `CompanionEngine` | 核心调度器 | 评分算法(0-100)、定时触发、上下文触发、用户主动触发、离线恢复 |

**关键算法 — 关怀评分（0-100）**：
```
总分 = 基础分 + 时间权重 + 情绪权重 + 事件权重 + 用户偏好权重
```
当评分超过阈值时触发关怀交互。

**内置调度器**：
- `start_scheduler()` / `stop_scheduler()` — 30分钟间隔自动检查关怀时机
- 随应用启动，懒加载模式

## 关键技术特性

### 1. 防骚扰机制 🛡️
| 策略 | 参数 | 说明 |
|------|------|------|
| 每日预算限制 | 5次/天 | 超过预算当日不再主动触发 |
| 连续限制 | 2次 | 连续关怀不超过2次，避免轰炸 |
| 拒绝惩罚 | 冷却翻倍 | 用户拒绝后冷却时间加长 |
| 交互锁 | asyncio.Lock | 防止并发触发多个关怀 |
| 动态预算调整 | 根据反馈 | 用户接受率高则适当增加预算 |

### 2. 渐进式建档 📝
5步 ONBOARDING_SEQUENCE：
1. **基础信息** — 姓名、称呼偏好
2. **生活节奏** — 作息时间、工作类型
3. **兴趣爱好** — 娱乐偏好、学习兴趣
4. **家庭关系** — 家庭成员基本信息
5. **健康关注** — 健康状况、运动习惯

配合 5 条 INFERENCE_RULES 行为推断规则，从用户日常对话中自动推断属性。

### 3. 情绪感知系统 🎭
MoodDetector 通过关键词匹配检测 5 种情绪状态：
| 情绪 | 关键词示例 | 关怀策略 |
|------|-----------|---------|
| positive | 开心、太好了、哈哈 | 分享喜悦，正向强化 |
| negative | 难过、烦、糟糕 | 安慰陪伴，情绪疏导 |
| neutral | — | 日常关怀，轻量互动 |
| stressed | 压力大、焦虑、deadline | 减压建议，情绪支持 |
| tired | 好累、困了、熬夜 | 休息提醒，健康关注 |

### 4. 用户主动触发 🗣️
用户可以通过 USER_CARE_REQUEST_KEYWORDS 中的关键词主动触发关怀：
- 匹配关键词后绕过冷却机制直接触发
- 确保用户需要时能立即获得关怀回应

### 5. EventBus 解耦架构 🔌
```
CompanionEngine ──publish──▶ EventBus ──subscribe──▶ GuiAgent
     (Core层)                (中间件)                  (UI层)
```
- 陪伴引擎通过 EventBus 发布事件，与 UI 层完全解耦
- UI 层订阅事件后自行处理消息展示和语音播报
- 支持未来扩展其他消费者（如远程通知、日志记录）

## 经验教训

### 1. 防骚扰设计至关重要
主动关怀系统最大的风险是"好心办坏事"——频繁打扰用户。必须在设计之初就建立完善的防骚扰机制：
- 每日预算硬限制是底线
- 用户拒绝反馈必须被尊重（惩罚机制）
- asyncio.Lock 防止并发导致的超额触发

### 2. 渐进式信息收集优于一次性表单
用户不喜欢填写冗长的信息表单。通过 ONBOARDING_SEQUENCE 分步收集 + INFERENCE_RULES 行为推断，可以在自然对话中逐步建立用户画像，用户体验更好。

### 3. EventBus 解耦的价值
将 CompanionEngine 与 UI 层通过 EventBus 完全解耦，带来了：
- 测试便利：可以独立测试引擎逻辑
- 扩展性：未来可以轻松添加远程通知等新消费者
- 维护性：UI 变更不影响引擎逻辑

### 4. 情绪检测的局限性
当前基于关键词匹配的情绪检测方案简单但有局限：
- 无法理解上下文语义（如反语、玩笑）
- 中文情绪表达多样，关键词覆盖有限
- 未来可考虑引入 LLM 情绪分析增强
